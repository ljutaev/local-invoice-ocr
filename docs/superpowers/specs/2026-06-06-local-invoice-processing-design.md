# Локальна обробка інвойсів — дизайн-документ

**Дата:** 2026-06-06
**Статус:** на рев'ю
**Залізо:** MacBook Pro M4 Pro · 24 ГБ unified memory · macOS 15.1

---

## 1. Мета і контекст

Система локально обробляє інвойси (рахунки): зчитує документ, витягує структуровані
дані, дає людині перевірити точність, зберігає результат у зашифрованій базі. Уся
обробка — **локальна**: жоден байт не залишає машину. Це сильніша приватність, ніж у
хмарних AP-сервісів (орієнтир користувача — «Reform», але без відправки в хмару).

**Що НЕ входить (поки):** інтеграція з конкретною зовнішньою системою (ERP/бухгалтерія) —
лишаємо точку розширення (`Exporter`), реалізація — пізніше.

### Підтверджені рішення
- **LLM-витягування:** локально через Ollama (не хмара).
- **Рушій читання:** Підхід A — двостадійний (текст із PDF або OCR → потім LLM).
- **Вхід:** суміш цифрових PDF + сканів/фото; мова — англійська.
- **БД:** SQLite (WAL).
- **Шифрування:** чутливі поля на рівні застосунку + зашифровані оригінали-файли.
- **Канали надходження:** тека-приймач + веб-завантаження + email.
- **Перегляд:** side-by-side (оригінал ↔ поля, підсвітка низької впевненості, редагування).
- **Обсяг:** сотні+ інвойсів/день.

## 2. Архітектура

```
   ┌─ тека-приймач ─┐
   ├─ веб-upload ───┤   адаптери надходження → спільний інтерфейс Ingest
   └─ email (Ф.3) ──┘
          │  hash+дедуп, шифрує оригінал, job(status=pending)
          ▼
   ┌──────────────┐  черга в SQLite: BEGIN IMMEDIATE +
   │    Queue     │  UPDATE ... RETURNING (захоплення job)
   └──────┬───────┘
          ▼  worker (пул, LLM-семафор=1–2)
   ┌─────────────────────────────────────────────┐
   │ Reader     PDF з текстом → PyMuPDF            │
   │            скан/фото    → Tesseract OCR (+boxes)│
   ├─────────────────────────────────────────────┤
   │ Extractor  текст → Ollama → строгий JSON       │
   │            за схемою інвойсу (format=schema)   │
   ├─────────────────────────────────────────────┤
   │ Validator  правила → впевненість per-field     │
   └──────┬──────────────────────────────────────┘
          ▼
   ┌──────────────┐  SQLite + шифрування чутливих полів,
   │    Store     │  оригінали AES-256-GCM, ключ у Keychain,
   └──────┬───────┘  audit_log
          ▼ status=needs_review
   ┌──────────────┐  FastAPI + PDF.js (localhost)
   │  Review UI   │  оригінал ↔ поля, підсвітка, редагування, approve
   └──────┬───────┘
          ▼ status=verified
   ┌──────────────┐  інтерфейс-заглушка; поки: CSV/JSON-експорт
   │   Exporter   │  (точка розширення під ERP/API)
   └──────────────┘
```

## 3. Компоненти (межі, входи/виходи, залежності)

Кожен компонент має одну відповідальність, спілкується через явний інтерфейс, тестується
окремо.

### 3.1 `ingest` — приймання документів
- **Що робить:** приймає файл (PDF/PNG/JPG) з джерела, рахує SHA-256 (дедуп), шифрує
  оригінал на диск, створює `job(status=pending)`.
- **Адаптери (спільний інтерфейс `Source`):** `FolderSource` (watch теки), `UploadSource`
  (через Review UI), `EmailSource` (IMAP, Фаза 3).
- **Вхід:** байти файлу + метадані джерела. **Вихід:** `job_id`.
- **Залежить від:** `store` (шифрування файлу), БД.

### 3.2 `queue` — черга задач (SQLite)
- **Що робить:** видає воркеру наступну pending-задачу атомарно.
- **Механізм:** `BEGIN IMMEDIATE; UPDATE jobs SET status='processing', worker_id=?,
  started_at=? WHERE id=(SELECT id FROM jobs WHERE status='pending' ORDER BY created_at
  LIMIT 1) RETURNING id; COMMIT;` (SQLite ≥3.35). WAL-режим, `busy_timeout`.
- **Вхід:** — **Вихід:** `job` або «порожньо».

### 3.3 `reader` — документ → текст (+боксы)
- **Що робить:** маршрутизує: якщо PDF має текстовий шар (перевірка PyMuPDF) → прямий
  витяг тексту; інакше (скан/фото/image-only PDF) → растеризація сторінок → Tesseract OCR
  (eng). Повертає текст по сторінках + bounding boxes слів (для підсвітки в UI).
- **Вхід:** дешифрований файл (в пам'яті). **Вихід:** `ReaderResult{pages:[{text, boxes}]}`.
- **Залежить від:** PyMuPDF, pytesseract. Чистий/детермінований — легко тестувати.

### 3.4 `extractor` — текст → структурований JSON
- **Що робить:** надсилає текст у локальну Ollama-модель зі **строгою JSON-схемою**
  інвойсу (Ollama `format`=JSON schema, щоб гарантувати валідний JSON).
- **Схема полів:** `invoice_number, invoice_date, due_date, vendor{name,address,tax_id},
  buyer{name}, currency, line_items[]{description,qty,unit_price,amount}, subtotal, tax,
  total, po_number, payment_terms`.
- **Модель:** `qwen2.5:14b` (баланс точність/RAM на 24 ГБ). Конфігурована.
- **Вхід:** текст. **Вихід:** `ExtractionResult{fields, raw_model_output}`.
- **Помилки:** невалідний JSON → повтор зі суворішим промптом; після N невдач → передає
  далі з прапором `extraction_failed` і сирим текстом для ручного розбору.
- **Залежить від:** Ollama HTTP API (localhost:11434).

### 3.5 `validator` — впевненість через перевірки (НЕ слова моделі)
- **Що робить:** обчислює per-field прапори впевненості правилами:
  - арифметика: `Σ line_items.amount == subtotal`, `subtotal + tax == total`;
  - дати/числа парсяться у валідні типи;
  - **grounding:** витягнуте значення дослівно (нормалізовано) присутнє в OCR-тексті
    → захист від галюцинацій;
  - обов'язкові поля не порожні.
- **Вхід:** `ExtractionResult + ReaderResult`. **Вихід:** `fields + field_flags{field:
  {confidence: high|low, reasons:[]}}`. Чистий — тестується на табличних кейсах.

### 3.6 `store` — шифроване сховище
- **БД:** SQLite (WAL). Таблиці:
  - `jobs(id, source, source_ref, file_hash, enc_file_path, status, attempts, error,
    created_at, started_at, finished_at)`
  - `invoices(id, job_id, status[needs_review|verified|failed], enc_fields BLOB,
    field_flags JSON, confidence_summary, created_at, verified_at, verified_by)`
  - `audit_log(id, invoice_id, action, field, enc_old, enc_new, user, ts)`
- **Шифрування:**
  - Майстер-ключ — у **macOS Keychain** (через `keyring`/`security`).
  - Оригінали-файли: **AES-256-GCM**, унікальний nonce/файл, зберігаються як зашифровані
    блоби на диску; шлях — у `jobs.enc_file_path`.
  - Чутливі дані (`enc_fields`, значення в audit) — шифруються тим самим ключем (AES-GCM),
    зберігаються як BLOB.
  - **У відкритому вигляді лишаються** лише несенситивні: `status`, дати створення,
    `field_flags` (прапори впевненості — не самі значення). Це дозволяє списки/фільтри в UI
    без розшифрування всього.
  - **Компроміс:** агрегації SQL по сумах неможливі (суми зашифровані) — для поточної
    задачі (перевірка) це прийнятно; зведення рахуються в застосунку після розшифрування.

### 3.7 `reviewapp` — side-by-side перевірка
- **Що робить:** локальний веб-застосунок (FastAPI + PDF.js, лише `localhost`).
  - **Список:** інвойси з фільтром за статусом (needs_review / verified / failed).
  - **Деталь:** оригінал документа (PDF.js / `<img>`) ліворуч; форма витягнутих полів
    праворуч; поля з `low confidence` підсвічені; клік по полю → підсвітка регіону в
    документі (за OCR-боксами); редагування + **Approve**.
  - **Approve** → `status=verified`, запис у `audit_log`.
  - Розшифрування полів — в пам'яті застосунку лише для показу.
- **Залежить від:** `store`.

### 3.8 `exporter` — вивід (заглушка)
- **Що робить:** інтерфейс `Exporter.export(invoice)`; поки реалізація — CSV/JSON-експорт
  верифікованих інвойсів. Точка розширення під майбутній ERP/API.

### 3.9 `worker` — оркестратор
- Бере job з `queue` → `reader` → `extractor` → `validator` → `store` (status=needs_review).
- LLM — вузьке місце: семафор обмежує паралельні виклики Ollama (1–2); решта стадій можуть
  бути паралельні. Пул воркерів конфігурований.
- Помилки стадій → `job.status=failed` + текст; retry-політика (N автоповторів для
  транзієнтних; ручний retry-кнопка в UI).

---

## 4. Потік даних (повний цикл)

1. Файл надходить (тека/upload/email) → `ingest`: hash, дедуп, шифрує оригінал, `job=pending`.
2. `worker` захоплює job → `reader` дає текст(+боксы).
3. `extractor` → структурований JSON; `validator` → прапори впевненості.
4. `store` шифрує й зберігає → `invoice.status=needs_review`.
5. Людина в `reviewapp` перевіряє (підсвічені слабкі поля), править, **Approve** →
   `status=verified` + `audit_log`.
6. (Майбутнє) `exporter` віддає верифіковані дані в зовнішню систему.

---

## 5. Обробка помилок
- Дедуп за `file_hash` — повторний файл позначається, не дублюється.
- `reader`: немає тексту/OCR провалився → job=failed, видно в UI для ручного розбору.
- `extractor`: невалідний JSON → повтор зі суворішим промптом → після N → needs_review із
  сирим текстом.
- `validator`: усі слабкі поля підсвічуються (не блокують, але вимагають уваги людини).
- Транзієнтні (Ollama недоступна) → автоповтор з backoff; постійні → failed + ручний retry.

## 6. Безпека / приватність
- **Локальність:** обробка повністю на машині; мережево — лише localhost (Ollama, Review UI).
- **At rest:** оригінали + чутливі поля зашифровані (AES-256-GCM); ключ у Keychain.
- **Аудит:** `audit_log` фіксує хто/що/коли правив (значення — зашифровані).
- **Доступ:** Review UI слухає лише `127.0.0.1`; для віддаленого доступу — лише через
  VPN/тунель (поза скоупом MVP).
- Ключі/секрети не потрапляють у git.

## 8. Тестування (TDD)
- **Юніт:** маршрутизатор `reader`; обгортка OCR; валідація JSON-схеми `extractor`; правила
  `validator` (табличні кейси: збіг/незбіг арифметики, галюцинація поза текстом); round-trip
  шифрування `store`; захоплення job у `queue` (конкурентність).
- **Золотий набір:** кілька зразкових інвойсів (цифрові PDF + скани) з еталонним JSON →
  регресія точності витягування.
- **Інтеграційний:** повний цикл ingest→verified на тестовій теці.

## 9. Фази впровадження
- **Фаза 1 (MVP):** `FolderSource` ingest → `reader` → `extractor` → `validator` → `store`
  (SQLite, шифрування) + мінімальний CLI/список для підтвердження, що дані лягають у БД.
- **Фаза 2:** повний `reviewapp` (side-by-side, підсвітка, редагування, approve, audit) +
  `UploadSource`.
- **Фаза 3:** `EmailSource` (IMAP) + `exporter` (CSV/JSON) + гачки під ERP/API.
